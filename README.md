## サービス概要
パンの残り数と消費ペースを管理できるアプリです。
冷凍や購入のタイミングを迷わず判断できるようサポートします。
日常の小さな「考える手間」を減らし、パンを無駄にしない生活を目指します。

## このサービスへの思い・作りたい理由
私は昔から朝食はパン派です。
食パンを1日1枚ずつ食べていくと、夏場は冷凍しない限りカビが生えてしまいます。
パンを食べようとしたときにカビが生えていると、どうしても悲しい気持ちになります。

しかし、食パンを小分けにして冷凍する作業は意外と面倒で、
「あとでやろう」と思っているうちに後回しにしてしまうことが多くありました。

また、買い物の際に
「今、食パンはあと何枚残っていたか」
「今日は買うべきか、それとも次回で良いか」
と悩むことがよくあります。
こうした在庫確認や購入判断は小さなことですが、日常の中では意外とストレスになります。

もしパンの残り枚数や消費ペースをアプリで簡単に確認できれば、
冷凍するタイミングや購入の判断を迷わずに済み、
カビの生えたパンにショックを受けることも減らせるのではないかと考えました。

一人暮らしだけでなく、家族がいる場合は消費ペースも変わり、
パンを買うタイミングにはより判断が必要になります。
そうした「考える手間」を減らし、日常を少し楽にしたいと思い、
このサービスを作ろうと考えました。

## ユーザー層について
- パンを購入する人
- 一人暮らし、または家族・同居人と生活している人
-  パンの在庫や消費ペースを感覚で管理している人
本サービスは、
「きっちり管理する人」ではなく、
パンの購入判断や管理を楽にしたい人を対象としています。
日常的にパンを食べているものの、在庫管理や冷凍の判断を感覚で行っている人ほど、
このサービスによる負担軽減の効果が大きいと考えています。

## サービスの利用イメージ
ユーザーはパンを購入したタイミングで、
パンの入り数と消費期限を入力するだけで利用を開始できます。

何枚切りか、1日に何枚消費するかは家庭によって異なるため、
初回に個人設定として登録し、以降はデフォルトで入力されるようにします。
これにより、毎回細かく入力する手間を省きます。

消費期限が近づいたタイミングで通知が届くため、
冷凍や消費の判断を迷わずに行うことができます。
また、登録後はトップ画面に
「残りのパンの枚数」と「消費期限」が大きく表示されるため、
買い物中でもアプリを見るだけで在庫状況を把握できます。

これにより、パンの残りの数を気にしながら買い物をする必要がなくなり、
日常の小さなストレスを減らすことができます。


## サービスの差別化ポイント・推しポイント
一般的な在庫管理アプリは、管理できる項目や機能が多く、
使い続けるうちに入力が負担になってしまうことがあります。

本サービスは「パン」に対象を絞ることで、
ズボラな人でも迷わず使えるシンプルさを重視しています。

登録作業は最小限に抑え、
一度設定した内容はデフォルト入力されるため、
ユーザーに余計な負担をかけません。

多機能ではなく、
「パンを買う・消費する・判断する」という行動に特化している点が
本サービスの最大の差別化ポイントです。


## 機能候補
### MVP
- ユーザー登録・ログイン機能
- トップ画面にパンの在庫と消費期限を表示
- 消費期限に基づく通知機能
  - 画面内通知
  - ブラウザ通知
- 消費枚数から残り日数を自動計算するロジック
- 初期設定値を自動入力するUX設計
### 本リリース
  - 家族間で情報を共有できるグループ機能（本リリース予定）
  - 消費期限をカメラで読み取るOCR機能（本リリース予定）
  - LINEと連携した通知機能（本リリース予定）

## 使用する技術スタック
| カテゴリ        | 技術                                    | 選定理由                                                                   |
| ----------- | ------------------------------------- | ---------------------------------------------------------------------- |
| バックエンド      | Ruby on Rails 7.1.4 / Ruby 3.2.2      | CRUDに加えて通知などの処理まで一貫して実装できるため。学習中の技術であり、理解した上で安定した開発が可能なため。 |
| フロントエンド     | JavaScript / Hotwire（Turbo, Stimulus） | Rails標準技術を利用することで、フロントとバックエンドを分断せずに開発できるため。部分的な画面更新によりUX向上が図れるため。      |
| CSSフレームワーク  | Tailwind CSS / DaisyUI                | コンポーネントベースで直感的にUIを構築でき、ポップで分かりやすい画面を短時間で実装できるため。                       |
| データベース      | PostgreSQL                            | 本番運用での実績が多く、信頼性が高いため。またRenderとの相性が良く、デプロイ構成がシンプルになるため。                 |
| 認証          | Devise                                | 安全なユーザー認証機能を短時間で実装でき、将来的な家族共有機能の基盤として利用できるため。                          |

| 外部API連携     | LINE Messaging API（本リリース予定）           | 日常的に利用されているLINEで通知を行うことで、ユーザーが気付きやすく、家族間での共有にも適しているため。                 |
| 文字認識（OCR）   | rtesseract（本リリース予定）                   | 消費期限入力の手間を減らし、ズボラなユーザーでも使いやすいUXを実現するため。                                |
| インフラ / デプロイ | Render                                | RailsとPostgreSQLの実績が多く、個人開発でも安定した本番運用が可能なため。                           |
| 環境構築        | Docker                                | 開発環境差異によるトラブルを防ぎ、誰でも同じ環境を再現できるようにするため。                                 |
| CI/CD       | GitHub Actions                        | push時に自動でテストを実行し、品質を保ちながら開発を進めるため。                                     |
